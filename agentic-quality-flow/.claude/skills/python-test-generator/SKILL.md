# Tier2 Python Test Generator Skill

## Purpose

Generates **working tier2 Python/pytest test code** from STD YAML specifications.

**Output:** Working Python test files that pass `pytest --collect-only`

**Note:** For test stubs (design review), use `python-stub-generator` instead.

**Critical:** Follows patterns from comprehensive LSP analysis of 290+ tier2 test files.

---

## CRITICAL REQUIREMENT

**Generate ONE test function per STD scenario. No exceptions.**

- ✅ CORRECT: 8 STD scenarios → 8 generated `def test_*()` functions
- ❌ WRONG: 8 STD scenarios → 3 test files (grouped without covering all scenarios)

**Pattern-based file grouping is allowed**, but **EVERY scenario must get a test function**.

Each test function must:
- Have a unique name derived from scenario test_id
- Include scenario ID in docstring for traceability
- Implement all steps and assertions from STD scenario

---

## Input Required

- `jira_id`: Jira ticket ID (e.g., "CNV-66855")

**Prerequisites:**
- STD YAML file must exist at `outputs/std/{JIRA_ID}/{JIRA_ID}_test_description.yaml`
- LSP patterns file generated by ticket-context-analyzer (fresh per-ticket)

outputs/python-tests/{JIRA_ID}/

## Workflow

### Step 1: Read STD YAML

Load `outputs/std/{JIRA_ID}/{JIRA_ID}_test_description.yaml`

**Extract:**
- Total scenario count: `len(scenarios)`
- All tier2 scenarios (filter by `tier: "Tier 2"`)

### Step 2: Load Pattern Libraries

**Read patterns from fresh LSP analysis:**

1. **Read** `outputs/python-tests/{JIRA_ID}/{JIRA_ID}_lsp_patterns_tier2.yaml` (FRESH LSP analysis)
   - This file contains the LATEST patterns from actual repository code
   - Created by ticket-context-analyzer agent using LSP
   - Contains actual function signatures, imports, and usage patterns

2. **Read** `{project_context.config_dir}/patterns/tier2_patterns.yaml` (template selection logic)

### Step 3: Group Scenarios by Pattern

**CRITICAL: This step is for file organization only. ALL scenarios must still get test functions.**

For each scenario:
1. Detect patterns (localnet, NAD type, VM type, operations)
2. Select template based on priority rules
3. Assign scenario to a file group

Result: Map of `{file_name: [scenario1, scenario2, ...]}`

Example:
```
{
  "test_localnet_connectivity.py": [TS-012, TS-013, TS-014],
  "test_localnet_advanced.py": [TS-015, TS-016],
  "test_localnet_negative.py": [TS-017, TS-018, TS-019]
}
```

### Step 4: For Each File Group

Generate the test file with ALL scenarios in the group.

**For each scenario in the group:**

1. **Detect Patterns**
   - Check for localnet (CRITICAL: special handling required)
   - NAD type: bridge, flat_overlay, SR-IOV, macvtap
   - VM type: Fedora, RHEL, Alpine
   - Connectivity: ping, TCP
   - Storage: DataVolume, snapshot, resize
   - Migration: live migration, eviction strategy

2. **Generate Test Function**
   - Function name: `def test_{scenario_id_slug}():`
     - Example: `test_id: "TS-CNV66855-012"` → `def test_ts_cnv66855_012_dual_stack_connectivity():`
   - Docstring: Include scenario ID for traceability
   - Test body: Implement all steps and assertions from STD scenario

3. **Collect Requirements**
   - Track imports needed for this test function
   - Track fixtures needed for this test function
   - Track markers for this test function

**After all scenarios in group processed:**

4. **Select Template** (priority-based from `{project_context.config_dir}/patterns/tier2_patterns.yaml`)
   - Priority 1: `{project_context.config_dir}/templates/tier2/localnet_cudn_test.py.template` ⚠️ SPECIAL CASE
   - Priority 2: `{project_context.config_dir}/templates/tier2/network_parametrized_test.py.template` (dual stack)
   - Priority 3: `{project_context.config_dir}/templates/tier2/migration_test.py.template`
   - Priority 4: `{project_context.config_dir}/templates/tier2/storage_test.py.template`
   - Priority 5: `{project_context.config_dir}/templates/tier2/vm_lifecycle_test.py.template`
   - Priority 6: `{project_context.config_dir}/templates/tier2/network_connectivity_test.py.template`
   - Priority 7: `{project_context.config_dir}/templates/tier2/basic_vm_test.py.template` (default)

5. **Read Template**
   - Load selected template from `{project_context.config_dir}/templates/tier2/` directory
   - Reference tests are available at `{project_context.config_dir}/reference/tier2/`

6. **Populate File-Level Placeholders**
   - `{{JIRA_ID}}`, `{{STP_REFERENCE}}`, `{{PR_URL}}`
   - `{{FEATURE_NAME}}`, `{{feature}}`, `{{FeatureClassName}}`
   - `{{NAD_TYPE}}`, `{{NETWORK_NAME}}`, `{{NAD_NAME}}`
   - `{{TIER}}`, `{{PRIORITY}}`, `{{POLARION_ID}}`
   - `{{TEST_FUNCTIONS}}` → ALL generated test functions from step 2
   - `{{ADDITIONAL_MARKERS}}`, `{{FIXTURES}}`

7. **Generate Imports**
   - **PRIORITY:** Use imports from `{JIRA_ID}_lsp_patterns_tier2.yaml` if available
   - **FALLBACK:** Use imports from `docs/tier2_pattern_library.yaml`
   - Follow LSP-validated import order
   - Example: `from tests.network.libs import cluster_user_defined_network as libcudn`

8. **Generate Fixtures**
   - **PRIORITY:** Use function signatures from `{JIRA_ID}_lsp_patterns_tier2.yaml` if available
   - **FALLBACK:** Use signatures from `docs/tier2_pattern_library.yaml`
   - Class-scoped for shared resources (VMs, NADs)
   - Function-scoped for test-specific resources
   - **CRITICAL:** Follow fixture naming convention: `{resource}_scope_{class|function}`
   - **CRITICAL:** Use EXACT function signatures from LSP analysis (don't invent parameters)

9. **Generate Class Docstrings**
   - **MANDATORY:** Include Markers section
   - Format from LSP analysis (see patterns/pattern_rules.yaml)
   - Example:
     ```python
     """
     Tests for localnet connectivity.

     Markers:
         - tier2
         - p0

     Preconditions:
         - Localnet network configured
     """
     ```

10. **Validate Generated Code**
    - **NEW:** Verify all function calls match LSP pattern signatures
    - **NEW:** Verify all imports match LSP pattern imports
    - Context managers used for all resources
    - `running_vm()` called after VM creation (or removed if wrapper handles it)
    - Class docstrings include Markers section
    - No global fixture redefinition
    - **CRITICAL:** Verify number of `def test_*()` functions equals number of scenarios in group

11. **Save File**
    - Derive filename: `test_{feature_slug}.py`
    - Save to `outputs/python-tests/{JIRA_ID}/test_{feature_slug}.py`

### Step 5: Generate conftest.py (if needed)

**Generate conftest.py ONLY IF:**
- Multiple test files share fixtures
- Module-scoped fixtures needed

**DO NOT include in conftest.py:**
- `admin_client` (already in tests/conftest.py)
- `unprivileged_client` (already in tests/conftest.py)
- `namespace` (usually test-specific)
- Any other global fixtures from main tests/conftest.py

**Include in conftest.py:**
- Test-specific shared fixtures (class/module scope)
- Constants used across multiple test files

### Step 6: Validate Complete Coverage

**CRITICAL VALIDATION - This step is MANDATORY**

After all files generated:

1. **Count STD scenarios:**
   - Count all Tier 2 scenarios in STD: `N_std`

2. **Count generated test functions:**
   - Count all `def test_*()` functions in generated files: `N_tests`

3. **Verify completeness:**
   - If `N_tests < N_std`:
     - ERROR: "Incomplete coverage - {N_std - N_tests} scenarios missing"
     - List missing scenario IDs
     - FAIL generation
   - If `N_tests == N_std`:
     - SUCCESS: "Complete coverage - all {N_std} scenarios have tests"

4. **Scenario ID mapping:**
   - For each STD scenario ID, verify it appears in test function docstrings
   - Report any missing IDs

### Step 7: Report Results

Generate summary with:
- Scenarios processed
- Python test files generated
- Conftest files generated
- Total lines of code
- Any errors or warnings

## Critical Pattern: Localnet Networks

**⚠️ CRITICAL:** Localnet networks use CUDN, NOT network_nad().

**WRONG (will fail with KeyError):**
```python
with network_nad(
    nad_type="localnet",  # ❌ KeyError: 'localnet'
    ...
) as nad:
```

**CORRECT (CUDN approach):**
```python
# Step 1: NNCP for bridge mappings (module scope)
@pytest.fixture(scope="module")
def nncp_localnet(admin_client):
    with libnncp.NodeNetworkConfigurationPolicy(...) as nncp:
        yield nncp

# Step 2: CUDN for localnet (class scope)
@pytest.fixture(scope="class")
def localnet_cudn_fixture(admin_client, nncp_localnet):
    with localnet_cudn(name="...", vlan_id=100, ...) as cudn:
        yield cudn

# Step 3: VM using localnet_vm() helper
@pytest.fixture(scope="class")
def vm_localnet(unprivileged_client, namespace, localnet_cudn_fixture):
    with localnet_vm(...) as vm:
        yield vm

# Step 4: IP extraction
def vm_ip(vm_localnet):
    return lookup_iface_status_ip(vm=vm_localnet, iface_name="...", ip_family=4)
```

**Detection:** If scenario contains "localnet" keyword → Use `templates/localnet_cudn_test.py.template`

---

## Critical Pattern: Fixture Naming

**Follow LSP convention:**
```python
# ✅ CORRECT
@pytest.fixture(scope="class")
def vm_for_migration_scope_class(namespace):
    ...

# ❌ WRONG (no scope suffix)
@pytest.fixture(scope="class")
def vm_for_migration(namespace):
    ...
```

**Pattern:** `{resource}_scope_{function|class|module|session}`

---

## Critical Pattern: Class Docstrings

**⚠️ MANDATORY:** Class docstrings MUST include Markers section.

**✅ CORRECT:**
```python
class TestFeature:
    """
    Tests for feature X.

    Markers:
        - tier1
        - p0
        - s390x

    Preconditions:
        - Running VM
    """
```

**❌ WRONG (missing Markers section):**
```python
class TestFeature:
    """
    Tests for feature X.

    Preconditions:
        - Running VM
    """
```

---

## Critical Pattern: Context Managers

**✅ CORRECT:**
```python
with VirtualMachineForTests(...) as vm:
    running_vm(vm=vm)  # ✅ Call running_vm
    yield vm
# Automatic cleanup
```

**❌ WRONG:**
```python
vm = VirtualMachineForTests(...)
yield vm  # ❌ No context manager, no running_vm call
```

---

## Critical Pattern: Module-Level Markers

**Add pytestmark at module level:**
```python
pytestmark = pytest.mark.usefixtures("namespace")

# Or multiple:
pytestmark = [
    pytest.mark.usefixtures("namespace"),
    pytest.mark.tier2,
]
```

## Error Handling

**If STD file not found:**
- Error + suggestion to run `/std-builder {JIRA_ID}` first
- Exit

**If LSP patterns file not found:**
- Error: "LSP patterns file not found. Run /generate-python-tests which spawns ticket-context-analyzer first."
- Exit

**If scenario pattern not recognized:**
- Warning + fallback to basic_vm_test.py.template
- Continue

**If validation fails:**
- Save to `.py.invalid` file
- Show validation errors
- Continue with remaining scenarios
